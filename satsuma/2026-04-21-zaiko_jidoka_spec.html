背景・現状の課題

1.1 在庫一覧表の概要

弊社では投資用不動産の仕入れ・販売業務において、社内の在庫一覧表（Googleスプレッドシート）を中心にすべての物件情報を管理している。在庫一覧表は以下の2つの領域で構成されており、物件の基本情報から各ポータルサイトへの掲載状況まで一元管理している。

📋 対象スプレッドシート： 在庫一覧表を開く →

列範囲区分主な管理項目
A列 〜 M列 案件の基本情報 営業担当 / BO担当 / 物件No / 所在地 / 売値 / 物件種別 / 築年数 / 想定利回り / 現況利回り / 物件詳細URL / 内見方法 / 内見可否 / みらいえ登録番号
N列 以降 ポータルサイト掲載状況 各ポータルサイトの掲載ステータス（掲載中 / 掲載不要 / 掲載中リンク）・ハイパーリンクURL

1.3 現状の課題

現状、在庫一覧表の管理はすべて手作業で行われており、以下の課題が発生している。

ISSUE 01

手作業の入力負担

各ポータルサイトを手動で確認し、掲載URLや掲載ステータスを1件ずつ在庫表に入力している。物件数が増えるにつれ作業量が増大し、担当者の業務を圧迫している。

ISSUE 02

リンク切れの見落とし

掲載URLのリンク切れ（404エラー等）を手動でチェックする仕組みがなく、気づかないまま古いURLが在庫表に残り続けるケースが発生している。

ISSUE 03

掲載ミス・入力ミス

手動入力のため、ポータルサイトと在庫表の情報の不一致（掲載中なのに未記入、掲載不要なのにURLが残っている等）が散発している。

ISSUE 04

掲載文の品質管理不足

各ポータルに掲載している物件紹介文の品質（誤字・訴求力・マーケティング表現）を体系的にチェックする仕組みがなく、品質のばらつきが生じている。

システム概要

2.1 目的

本システムは、在庫一覧表の管理業務を自動化することにより、以下を実現する。

  • 各ポータルサイトから自社物件の掲載URLを自動取得し、在庫一覧表に自動挿入する
  • 掲載URLのリンク切れを自動検知し、アラートで通知する
  • 掲載文の品質（誤字・訴求力）をAIで自動評価し、問題のある物件を担当者に通知する
  • 手作業の入力工数をゼロに近づけ、担当者が本来業務に集中できる環境を作る

2.2 全体フロー

本システムは以下の6つのフェーズで構成される。

↓

2
中間シートへの書き込み（NEMO08など専用シートを作成しリアテクスを抽出するイメージ）

スクレイピング結果をGoogleスプレッドシートの「スクレイピング結果」シートに保存・リンク切れも同時チェック

↓
Phase
3
在庫表との照合

所在地・価格・物件種別をキーにスコアリングで在庫表の各行と照合し、物件Noを紐付ける

↓
Phase
4
URLの自動挿入

照合済み物件の掲載URLを在庫表の該当ポータル列にハイパーリンクで自動挿入・掲載ステータスも更新

↓
Phase
5
AI品質チェック

機能仕様

Phase 1：スクレイピング（既存機能）

本フェーズは既存のスクレイピング基盤を使用する。新規実装は不要。

項目内容
検索方法各ポータルサイトで売主名「株式会社リアテクス」または「リアテクス」で検索
取得対象自社掲載物件の物件タイトル・所在地・価格・物件種別・掲載URL
対象サイト楽待 / 健美家 / HOMES / at home / SUUMO / 不動産連合体 / ジモティ
出力形式JSON または CSV（Phase 2 が読み込める形式）

Phase 2：中間シートへの書き込み（NEMO08など専用シートを作成しリアテクスを抽出するイメージ）

スクレイピング結果を保存するための中間シート「スクレイピング結果」をスプレッドシートに新設し、Pythonで書き込む。

No列名内容・備考
1取得日時スクレイピング実行日時（YYYY/MM/DD HH:MM）
2ポータル名楽待 / 健美家 / athome / suumo 等
3物件タイトルポータルサイト上の掲載タイトル
4所在地（スクレイプ）ポータルサイト上の所在地テキスト
5価格掲載価格（文字列）
6物件種別戸建 / アパート / マンション 等
7掲載URL物件詳細ページのURL
8リンク状態HTTPステータス確認結果（OK / リンク切れ / エラー）
9照合ステータス照合済 / 要確認 / 未照合
10照合先物件No在庫表の物件Noと紐付けた値
11AIスコアPhase 5のAI評価スコア（0〜100点）
12判定✅ / 🟡 / 🔴
13誤字チェック結果検出された誤字・表現ミスの一覧
14マーケティングコメントAIが生成した販売マーケティングコメント（100〜200文字）
15改善提案AIが提案する改善点（箇条書き）
16アラートメッセージ担当者向けの注意メッセージ

Phase 3：在庫表との照合

スクレイピング結果と在庫一覧表の行を照合し、同一物件を特定する。住所の表記ゆれに対応するため、スコアリング方式で照合する。

優先度照合キー配点備考
1位所在地の部分一致60点スクレイプ住所に在庫表住所の主要部分が含まれているか
2位価格の一致30点万円単位で一致しているか（表記ゆれを吸収して比較）
3位物件種別の一致10点戸建 / アパート / マンション の大分類が一致しているか
合計スコア照合結果処理
70点以上照合済在庫表に自動でURLを挿入する
40〜69点要確認「要確認」フラグを立て、担当者が手動で確認する
39点以下未照合新規物件の可能性あり。担当者に確認を促す

Phase 4：URLの自動挿入

照合済みの物件に対し、在庫一覧表の該当列に掲載URLをハイパーリンク形式で自動挿入する。

ポータルサイト在庫表への書き込み列書き込み内容
SUUMO自動返信/SUUMO 列掲載URLをハイパーリンクで挿入・ステータスを「掲載中」に更新
健美家・HOMES自動返信/健美家・homes 列同上
at home自動返信/at home 列同上
楽待不可/楽待 列同上
不動産連合体自動返信/不動産連合体 列同上
ジモティ自動返信/ジモティ 列同上
⚠ リンク切れ（HTTP 404）が検出された場合は、該当セルを赤背景にし、URLの隣に「⚠ リンク切れ」と表示する。

Phase 5：AI品質チェック

在庫一覧表に挿入されたURLのページを取得し、Claude APIを使って掲載文の品質を自動評価する。

① 誤字・表現チェック
20点満点

漢字ミス・送り仮名・数字の表記ゆれ・不自然な文章の有無を検出する

② マーケティング訴求チェック
80点満点

投資家の購買意欲を引き出す訴求か・数字の使い方・キャッチコピーの具体性を評価する

AI出力項目（JSON形式）

フィールド名型内容
total_scoreint総合スコア（0〜100）
typo_check.has_issuesbool誤字・表現ミスの有無
typo_check.issueslist検出された誤字・表現ミスのリスト
marketing_check.scoreintマーケティング評価スコア（0〜80）
marketing_check.is_content_insufficientbool内容が著しく不十分かどうか
marketing_check.commentstrマーケティングコメント（100〜200文字）
improvement_suggestionslist改善提案のリスト
rewrite_neededbool紹介文の再作成が必要かどうか

Phase 6：アラート判定・シートへの反映

AI評価の結果に基づいてアラートレベルを判定し、スプレッドシートに反映する。

判定条件アラートメッセージ
🔴 要対応 AIスコアが80点以下 「AIスコアが{score}点です。掲載文の改善を推奨します」
🔴 要対応 コメント内容が不十分（is_content_insufficient = true） 「訴求内容が不十分です。紹介文の再作成を推奨します」
🔴 要対応 誤字・表現ミスが検出された（has_issues = true） 「誤字が検出されました：{誤字の内容}」
🔴 要対応 リンク切れ（HTTP 404等） 「リンク切れを検出しました。URLを確認してください」
✅ 問題なし 上記いずれにも該当しない アラートなし
🔴 が1件以上発生した物件は、在庫一覧表の該当行の判定列に🔴を表示する。担当者は🔴の案件を優先して紹介文の修正を行う。

技術仕様

4.1 使用技術・ライブラリ

種別ライブラリ／サービス用途
言語Python 3.10 以上メインの実装言語
AIモデルClaude API（anthropic SDK）掲載文の品質評価
スプレッドシート連携gspread + google-authGoogleスプレッドシートの読み書き
Webスクレイピングrequests + BeautifulSoup4 + lxmlポータルサイトからのテキスト抽出
設定管理python-dotenvAPIキー等の環境変数管理
スクレイピング基盤既存スクレイパー（流用）各ポータルサイトの物件URL取得

4.2 ファイル構成

quality_checker/
  ├── main.py  # 全体を統括して実行するエントリーポイント
  ├── config.py  # スプレッドシートID・列マッピング・閾値などの設定
  ├── page_fetcher.py  # URLを取得しテキストを抽出する
  ├── ai_reviewer.py  # Claude APIで誤字・マーケティング評価を実行
  ├── alert_generator.py  # アラートレベルと通知メッセージを生成
  ├── sheet_manager.py  # スプレッドシートの読み書き
  ├── requirements.txt  # 依存ライブラリの一覧
  ├── service_account.json  # Google Sheets API認証キー（要発行）
  └── .env  # ANTHROPIC_API_KEY 等（Gitに含めないこと）

4.3 認証・APIキーの設定

  • Anthropic APIキー：環境変数 ANTHROPIC_API_KEY に設定する
  • Google Sheets API：GCPでサービスアカウントを作成し、service_account.json を配置する
  • スプレッドシートにサービスアカウントのメールアドレスを編集者として共有する
5

実行スケジュール・運用

5.1 推奨実行タイミング

実行タイミング実行頻度実行内容
毎朝（業務開始前）毎日Phase 1〜4（スクレイピング・URL挿入・リンク切れ検知）
週1回（月曜朝）週次Phase 5〜6（AI品質チェック・アラート）

5.2 実行方法

  • サーバー・PC上でcronジョブとして定期実行する
  • Google Cloud Scheduler + Cloud Run での実行も可能
  • 手動実行：python main.py コマンドで任意のタイミングで実行可能

将来的な拡張（参考）

本仕様書の範囲外だが、将来的に以下の拡張が考えられる。

  • アラート発生時にSlack・LINE WORKS・メールで担当者に自動通知する
  • AIが生成した改善提案をもとに紹介文の再作成案を自動生成する
  • kintoneとのリアルタイム双方向連携（在庫表の更新をkintoneに即時反映）
  • 掲載文のポータル別最適化（各サイトの特性に合わせた文章の自動調整）
